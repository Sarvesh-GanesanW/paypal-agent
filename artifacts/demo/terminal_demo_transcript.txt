## Two-Step Invoice Flow (PayPal Invoicing v2)

PayPal uses an intentional two-step design for invoicing: creation is separated from delivery so you can review, edit, or delete drafts before the payer ever sees them.

---

### Step 1: Create a Draft Invoice

**Endpoint:** `POST /v2/invoicing/invoices`

- Creates an invoice in **DRAFT** state.
- Your request body must include:
  - Merchant information (PayPal account in good standing)
  - An **`items` array** (required)
  - Other details as needed (payer info, amounts, due date, etc.)
- A draft invoice is **not visible or payable** by the payer — it exists only on your side for review/editing.
- On success, PayPal returns the invoice object with a generated ID (e.g., `INV2-XXXX-XXXX-XXXX-XXXX`).

---

### Step 2: Send the Invoice

**Endpoint:** `POST /v2/invoicing/invoices/{invoice_id}/send`

- Transitions the invoice from **DRAFT → SENT**.
- The payer receives an email notification and the invoice becomes payable.

---

### Why Two Steps?

> *"Creates a draft invoice. To move the invoice from a draft to payable state, you must **send** the invoice."*

This lets you safely build, review, and modify invoices before committing to deliver them.

---

### Invoice Status Lifecycle

| Status | Meaning |
|--------|---------|
| **DRAFT** | Not yet sent to payer |
| **SENT** | Awaiting payment |
| **SCHEDULED** | Will be sent on a future date |
| **PAID** | Payer has paid in full |
| **PARTIALLY_PAID** | Partial payment received |
| **MARKED_AS_PAID** | Manually marked paid by invoicer |
| **CANCELLED** | Cancelled by invoicer |
| **REFUNDED** | Refunded by invoicer |

---

## Safety Checks & LangSmith Tracing — Explained

Here's how this system prevents accidental live PayPal mutations and how LangSmith tracing supports debugging:

---

### 🔒 Safety Checks Preventing Accidental Live Mutations

| Layer | Mechanism | Description |
|-------|-----------|-------------|
| **Environment Isolation** | `PAYPAL_MODE=sandbox` env var | All API calls route to `api-m.sandbox.paypal.com` unless explicitly switched to `live`. The agent refuses mutating tools if the mode is unclear. |
| **Credential Separation** | Distinct `CLIENT_ID` / `CLIENT_SECRET` per environment | Sandbox credentials cannot authenticate against production endpoints and vice versa. |
| **Router Confirmation Gate** | Intent classification + missing-input detection | Before any mutating action (capture, refund, void, send invoice), the router verifies all required IDs/amounts are explicitly provided by you—never invented. |
| **Read-First Pattern** | GET before POST | For captures or refunds, the agent first calls a read endpoint (e.g., `GET /v2/checkout/orders/{id}`) to confirm the resource state, then asks you to confirm before executing. |
| **Idempotency Headers** | `PayPal-Request-Id` | Sent on every POST to prevent duplicate mutations if a retry occurs. |
| **Tool-Level Guards** | Schema validation | Each tool enforces required fields at the schema level; the LLM cannot bypass missing parameters. |

---

### 🔍 How LangSmith Tracing Helps Debug a Request

| Capability | Benefit |
|------------|---------|
| **End-to-End Trace Tree** | Every request produces a trace showing: Router → Tool Selection → API Call → Response Parsing → Final Answer. You can pinpoint exactly where a failure occurred. |
| **Latency Breakdown** | Each node shows wall-clock time, so you can identify whether a PayPal API call is slow vs. LLM inference. |
| **Token & Cost Tracking** | Input/output token counts per LLM call help optimize prompt length and cost. |
| **Input/Output Inspection** | View the exact payload sent to PayPal and the raw response—trivial to spot malformed requests or unexpected errors. |
| **Feedback & Annotation** | Traces can be marked correct/incorrect, building a regression-testing dataset before production rollout. |
| **Run Filtering** | Filter by metadata (e.g., `tool_name=capture_payment`, `environment=sandbox`) to audit all mutation attempts. |

---

### ✅ Bottom Line

- **You cannot accidentally hit production** unless you explicitly configure live credentials *and* set the environment to `live`.
- **Every mutating action** requires user-supplied IDs (never fabricated) and a confirmation step.
- **LangSmith tracing** gives full observability into every step, letting you catch issues in sandbox before promoting to production.
